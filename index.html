CLI 第三方 API 配置

Claude Code、Codex CLI、Gemini CLI
API中转站配置教程

面向 Windows 用户，按第三方 API 站点提供的 Base URL、API Key 和模型名生成配置。教程优先使用各 CLI 当前稳定的配置入口，并把容易失效的协议差异写清楚。

Base URL 填写到协议根路径 不要填完整请求端点。Codex/OpenAI 兼容通常填到 /v1；Claude/Anthropic 兼容通常填到 /v1/messages 前一级，具体以站点文档为准。
模型名必须用站点给出的原始值 不要自己猜模型名。CLI 只会把请求发出去，模型是否存在由第三方站点决定。
先小请求测试 配好后先执行一个简单提问，确认鉴权、协议和模型都可用，再放到正式项目里使用。

按顺序完成识别协议、查看步骤、生成配置、测试连通。内容会跟随当前 CLI 标签自动切换。

    Claude Code 连接 Anthropic 兼容站点

    Claude Code 最稳的第三方站点配置方式，是在用户级 settings.json 里写入环境变量。这样每次运行 claude 都会自动携带 Base URL、API Key 和模型名。

    配置文件：%USERPROFILE%\.claude\settings.json 协议：Anthropic / Claude 兼容

    站点兼容性

    第三方站点需要支持 Claude / Anthropic 风格接口。只提供 OpenAI 兼容接口的站点，不能直接按这页配置 Claude Code，除非站点明确提供 Claude 兼容转发。

    Windows 步骤

    1. 1
      创建配置目录

      PowerShell 执行：

      New-Item -ItemType Directory -Force "$env:USERPROFILE\.claude"
    2. 2
      打开用户级配置文件

      如果文件不存在，记事本会询问是否创建。

      notepad "$env:USERPROFILE\.claude\settings.json"
    3. 3
      粘贴右侧生成的 JSON

      保存后重新打开终端，让新环境变量生效。

    4. 4
      启动并测试
      claude

    排查点

    • 401 / Unauthorized：API Key、额度或站点分组权限不对。
    • 404 / model not found：模型名和站点后台不一致。
    • 连接失败：Base URL 路径不对，或站点不支持 Anthropic 兼容协议。

    配置生成器

    保存到当前浏览器本地；包含 API Key，建议仅在个人电脑使用。
    填到 Anthropic 协议根路径，不要填完整 endpoint。若站点文档给的是 https://example.com/anthropic/v1/messages，这里填 https://example.com/anthropic；若文档明确要求带 /v1，按站点文档填写。
    %USERPROFILE%\.claude\settings.json
    API Key 连通性测试
    使用当前 Base URL、API Key 和模型名发送最小 Messages 请求。浏览器测试可能被 CORS 拦截，PowerShell 命令结果更可靠。

    Codex CLI 连接 OpenAI 兼容站点

    Codex 的稳定配置入口是 %USERPROFILE%\.codex\config.toml。第三方站点需要兼容 OpenAI Responses API；如果站点只支持旧 Chat Completions，当前 Codex 可能无法稳定工作。

    配置文件：%USERPROFILE%\.codex\config.toml 密钥环境变量：OPENAI_API_KEY 协议：OpenAI Responses

    站点兼容性

    向站点确认它支持 /v1/responses。只写 /v1/chat/completions 兼容的站点，适合普通 OpenAI SDK，不一定适合 Codex CLI。

    Windows 步骤

    1. 1
      保存 API Key 到用户环境变量

      执行右侧生成的 PowerShell 命令后，重新打开终端。

    2. 2
      创建 Codex 配置目录
      New-Item -ItemType Directory -Force "$env:USERPROFILE\.codex"
    3. 3
      打开配置文件
      notepad "$env:USERPROFILE\.codex\config.toml"
    4. 4
      粘贴右侧生成的 TOML 并测试
      codex

    排查点

    • 启动后仍走官方账号：检查 model_provider 是否等于自定义 provider 名。
    • 404 / unsupported endpoint：第三方站点不支持 Responses API 或 Base URL 少了 /v1。
    • TOML 解析失败：模型名、URL 或 key 里有特殊字符时重新用生成器生成。

    配置生成器

    保存到当前浏览器本地；包含 API Key，建议仅在个人电脑使用。
    填到 OpenAI 兼容根路径，通常要带 /v1，不要填 /responses 或 /chat/completions。示例：https://example.com/v1。
    选择后会自动填写上下文窗口，并把自动压缩阈值设为 80%。
    PowerShell：写入 OPENAI_API_KEY
    %USERPROFILE%\.codex\config.toml

    不要手写 auth.json。第三方 API Key 放在环境变量里，配置文件只引用 env_key，更稳定也更少泄露。

    API Key 连通性测试
    使用当前 Base URL、API Key 和模型名发送最小 Responses 请求。若返回 404，通常说明站点不支持 Codex 需要的 Responses API。

    Gemini CLI 连接 Gemini 兼容站点

    Gemini CLI 的第三方站点支持取决于站点是否兼容 Google Gemini API。API Key 可用 GEMINI_API_KEY 注入；Base URL 需要站点和当前 CLI 版本支持对应环境变量。

    密钥环境变量：GEMINI_API_KEY Base URL：GOOGLE_GEMINI_BASE_URL 协议：Gemini API 兼容

    站点兼容性

    Gemini CLI 对第三方 Base URL 的支持比 OpenAI SDK 更依赖版本。配置后先运行 gemini --version 和一个简单提问；如果仍访问官方地址，优先检查 CLI 版本和站点说明。

    Windows 步骤

    1. 1
      安装或更新 Gemini CLI
      npm install -g @google/gemini-cli@latest
    2. 2
      写入第三方站点环境变量

      执行右侧生成的 PowerShell 命令，然后重新打开终端。

    3. 3
      测试模型
      gemini --model gemini-2.5-pro

    排查点

    • 401 / API key not valid：第三方站点 key 不可用，或 CLI 没读到新环境变量。
    • 仍跳官方登录：删除旧的 Gemini CLI 登录状态，或在新终端中确认 GEMINI_API_KEY 存在。
    • Base URL 无效：该 Gemini CLI 版本或站点不支持自定义 Base URL。

    配置生成器

    保存到当前浏览器本地；包含 API Key，建议仅在个人电脑使用。
    填第三方站点提供的 Gemini API 根地址，不要填具体模型或生成端点。是否带 /v1beta、/v1 取决于站点说明；Gemini CLI 对自定义 Base URL 也取决于当前版本支持。
    PowerShell：写入 Gemini 环境变量

    如果第三方站点文档要求的 Base URL 环境变量名不是 GOOGLE_GEMINI_BASE_URL，以站点文档为准替换。

    API Key 连通性测试
    使用当前 Base URL、API Key 和模型名发送最小 generateContent 请求。Gemini 第三方 Base URL 是否可用取决于站点和 CLI 版本。
    尚未测试。
